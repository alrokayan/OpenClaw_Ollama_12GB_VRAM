# Running an Agentic Vision-Language Model on Consumer Hardware: A Case Study in Local AI Orchestration on a 12 GB GPU

## Abstract

We document the design, implementation, and debugging of a fully local agentic AI system built on constrained consumer hardware — a single NVIDIA GeForce RTX 4070 with 12 GB of video memory. The system couples OpenClaw, a personal AI gateway, to a locally served vision-language model (Qwen3.5, 9B parameters) that drives an Android emulator through a Model Context Protocol (MCP) bridge and is controlled over Telegram. Our target hardware sits below the memory tier that the platform's own documentation describes as marginal, which forced a series of non-obvious engineering decisions around context-window budgeting, key-value cache sizing, the assignment of emulator rendering across a dual-GPU host, and configuration integrity. The work's principal contribution is not the system itself but a catalog of reproducible failure modes and their root causes, several of which contradict widely circulated advice. We give particular attention to a class of silent failures — configuration writes that report success while changing nothing — and trace one such failure, which cost the greatest debugging effort, to an interaction between Windows PowerShell's native-command argument handling and a command-line JSON interface. We argue that on constrained hardware, the dominant engineering challenge is not raw model capability but the correct and verifiable propagation of a small number of coupled numeric parameters from configuration through to hardware allocation.

## 1. Introduction

Large language models capable of tool use and visual perception have made local, agent-style automation newly plausible on personal hardware. The appeal is direct: an agent that runs entirely on one's own machine, sees a device screen, reasons about it, and acts on it, without sending data to a third party. The obstacle is equally direct: the models that do this well were sized for hardware that most individuals do not own.

This paper is a case study in bridging that gap on a single 12 GB GPU. We do not present a novel algorithm or model. We present, instead, the engineering reality of assembling existing components — a gateway, a locally served model, an emulator, a device-control bridge, and a messaging channel — into a working whole on hardware that the components were not tuned for, and we document what broke along the way.

The value of such a record is specific. Much of the practical knowledge required to run local models lives in forum posts, blog articles, and chat-assistant summaries of uneven reliability. During this project we repeatedly encountered advice that was plausible, widely repeated, and wrong for our situation. We therefore treat the failures as the primary subject matter. Each was hit in practice, diagnosed against primary sources, and resolved; we report them so that others need not rediscover them.

We deliberately exclude one commonly suggested technique from our recommendations. A frequent piece of advice for adjusting a locally served model's parameters is to author a custom model definition file that hard-codes the desired settings. We found this approach unsuitable for our purposes: it fragments the source of truth for configuration, obscures the coupling between parameters that must move together, and does not interact cleanly with the gateway's own configuration lifecycle, which actively rewrites certain values. Our system instead treats the gateway's validated configuration interface as the single authority for all model parameters. Where we mention the alternative at all, it is to explain why we did not adopt it.

## 2. System Architecture

The system comprises five components arranged in a pipeline. A messaging channel receives user instructions and returns results. A gateway process mediates between the channel and an agent runtime. The agent is backed by a locally served vision-language model. The agent reaches the outside world through two extension mechanisms: a device-control bridge exposed as an MCP server, and a skill that encodes a perception-reasoning-action loop over that bridge's tools. The device under control is an Android emulator running on the same host.

Two design decisions in this arrangement are dictated by the hardware and deserve statement up front.

First, the emulator's display is rendered on the host's *integrated* GPU, not on the discrete GPU that serves the model. The motivation is memory: on a 12 GB card every megabyte of video memory is contested, the model's footprint is the binding constraint, and so the discrete GPU's memory must not be spent on emulator rendering. The obvious way to achieve this — and the one most commonly recommended — is to render the emulator's display in software, on the CPU. We found that this does not work on our host: under software rendering the guest operating system boots and its display compositor runs, yet no pixels are ever produced, and the emulator window and every screenshot come back blank. Because the agent's entire purpose is to perceive the screen, a blank framebuffer is a total failure rather than a performance trade. The working solution exploits a second GPU that hosts of this class already have: the CPU's integrated graphics. We pin the emulator's rendering processes to the integrated GPU through the operating system's per-application GPU preference and render in hardware there. This produces a correct image while leaving the discrete GPU's memory entirely for the model — the same goal software rendering was meant to serve, reached by a route that actually yields a picture. The episode is another instance of the paper's recurring theme: the widely circulated advice — here, "render in software to conserve discrete-GPU memory" — was confidently wrong for this hardware, and the correct approach was found only by verifying against what the machine actually did.

Second, the model is served with a context window far smaller than it advertises. The model reports a maximum context of 262,144 tokens. This figure describes what the model was trained to handle, not what the hardware can hold. The memory required for the key-value cache — the store of intermediate attention state that grows with context length — scales linearly with the number of tokens. At the full advertised context, this cache alone exceeds the total memory of the card. The system therefore caps the operative context at 65,536 tokens, a figure chosen to fit alongside the model weights within 12 GB while leaving margin for compute buffers.

## 3. The Central Constraint: Memory Budgeting

The single most important quantity in this system is the amount of video memory consumed at inference time, and the single most important discipline is keeping that quantity below the physical ceiling. We make the arithmetic explicit because every subsequent decision follows from it.

The model weights, quantized to a four-bit scheme with mixed-precision refinements on sensitive tensors, occupy approximately 6.6 GB on disk and in memory. This leaves roughly 5 GB of the card's 12 GB for everything else, of which the key-value cache is by far the largest consumer. Empirically, the resident footprint of the loaded model is the sum of its weights and this cache: at the full advertised context the total reaches approximately 16 GB, which cannot fit; at the capped context it settles near 7.7 GB, which fits comfortably and leaves the model running entirely on the GPU.

The distinction between "fits" and "runs on the GPU" is not academic. When the requested allocation exceeds available video memory, the serving runtime does not reduce the request. It instead places part of the cache in system memory and processes those portions on the CPU. The model continues to function, and answers correctly, but throughput collapses. The observable signature is a processor split — a report that inference is running partly on CPU and partly on GPU — where a healthy configuration shows the model resident entirely on the GPU. Much of our diagnostic effort reduced to distinguishing a configuration that was merely accepted from one that was actually resident on the GPU. These are different facts, and only the second matters.

## 4. The Coupled-Parameters Problem

The system's correctness depends on three numeric parameters that describe context length, each owned by a different layer, which must be held in agreement.

The first is the serving runtime's allocation parameter, which determines how much key-value cache memory is actually reserved on the hardware. This is the number that must fit the card.

The second is the gateway's prompt budget, the length of prompt the gateway believes it may construct before it must compact or truncate history. If this exceeds the runtime's allocation, the gateway will build prompts larger than the hardware allocated, and the excess is silently discarded by the runtime.

The third is a per-model effective budget field that the gateway's own schema describes as the practical window it compacts against, distinct from the model's advertised native maximum. This field exists precisely to let the advertised maximum remain as metadata while a smaller practical figure governs runtime behavior.

The failure mode when these diverge is insidious because it is silent. If the gateway budgets against a larger figure than the runtime allocated, no error is raised. The prompt is constructed to the gateway's budget, handed to the runtime, and quietly cut down to the runtime's smaller allocation. The user observes degraded coherence — the model appears to forget the beginning of its context — with nothing in any log to explain it. During onboarding we observed exactly this: the gateway reported a budget sixteen times larger than the runtime had allocated.

Our stance on this coupling is conservative. We set all three parameters to the same value. This makes the configuration schema-complete, in that the effective-budget field is populated rather than left undefined, while eliminating any gap between what the gateway believes it has and what the hardware provides. An alternative exists — letting the advertised maximum float to its native value while only the effective budget and the runtime allocation are capped — but this reintroduces the possibility of divergence, and its correctness depends on the gateway honoring the effective-budget field over the advertised maximum for this specific serving backend. That dependency is a claim about runtime behavior that we were unwilling to trust without direct verification, given that the cost of being wrong is precisely the silent truncation the whole exercise aims to prevent. We therefore keep the numbers equal and treat the floating-maximum optimization as a separate change requiring behavioral proof before adoption.

## 5. The Configuration Lifecycle and Its Adversarial Repair Step

A repair utility supplied with the gateway warrants specific attention, because it actively works against the memory discipline described above.

The gateway includes a diagnostic-and-repair command that inspects the configuration and corrects perceived problems. Among its corrections, it raises the runtime allocation parameter back to the model's advertised native maximum, on the reasoning that a value below the native maximum is a mistake to be fixed. On adequate hardware this is helpful. On our hardware it is destructive: it undoes precisely the cap that keeps the model resident on the GPU, and the next time the model loads, it loads at the native context and spills to the CPU.

The system therefore treats this repair step as a hazard to be sequenced around rather than a service to be invoked freely. The configuration is validated before repair is ever run, because an invalid configuration causes the repair step to silently restore a previous known-good copy and discard all pending changes, saving the user's work aside under an obscure filename with no prominent error. After repair runs, the cap is re-applied. This re-application is essential, and — as the following section relates — its failure to take effect was the source of the project's most protracted debugging episode.

## 6. A Silent Configuration-Write Failure: Root-Cause Analysis

The system exhibited a persistent, reproducible fault: after a full setup, diagnostic checks reported that the runtime allocation was at the native maximum rather than the cap, the effective-budget field was unset, and the model had spilled to the CPU. Three distinct symptoms, but — as is often the case — one underlying cause.

The diagnosis proceeded through two incorrect hypotheses before reaching the correct one, and we recount all three because the path is instructive.

The first hypothesis attributed the fault to a stale loaded model. The reasoning: the configuration might be correct, but a model already resident in memory, loaded at the old context and held there by a keep-alive timer, would not pick up a configuration change until evicted and reloaded. This hypothesis was partly right — such a stale model does exist and does require eviction — but it was not the root cause, because it assumed the configuration had been written correctly, which inspection later showed was not reliably true.

The second hypothesis, advanced by a separate coding assistant working on the same problem, attributed the fault to the merge semantics of the configuration-write command. The claim was that the command used to update the protected list of models did not merge entries by identity as documented, leaving duplicate entries and failing to target the intended row, so that the re-application reported success while changing nothing. This hypothesis had the great virtue of explaining the "reports success, changes nothing" symptom directly. It had the defect of being contradicted by the platform's own documentation, which presents the merge command, used exactly as our system used it, as the supported and recommended method for updating that path, and states plainly that it preserves existing entries. We confirmed this against primary documentation rather than accept the claim.

The correct root cause was found by eliminating the shell from suspicion one layer at a time. The configuration-write command accepted its payload as a command-line argument containing JSON. On the host, that command ran under the system's default shell. When the JSON payload was passed as an argument through this shell to the underlying native executable, the shell stripped the quotation marks from the JSON before the executable received it. The executable therefore received malformed input — object keys and string values shorn of their delimiting quotes — which it could not parse. The command failed to apply the intended change. Depending on the exact invocation, it either raised a parse error or, in the merge path, applied nothing while reporting success.

Two properties of this failure made it hard to see. First, it was invisible on inspection of the source: the JSON written to disk was well-formed, and the command that read it read it correctly; the corruption occurred only at the boundary between the shell and the native executable, after every artifact a developer would inspect looked correct. Second, it affected only one configuration write in the entire system. Every other write in the setup routine passed its payload through standard input rather than as a command-line argument, and standard input does not cross the argument-parsing boundary where the corruption occurred. The single write that used a command-line argument was the model-list update — the very write responsible for applying the context cap. It was, therefore, the only write that silently failed, which is exactly why the context cap never took effect and the model always loaded at the native context.

We verified this diagnosis directly. The identical payload, written to a file with its quotes intact and confirmed correct on disk, was passed to the command as an argument and arrived at the executable with its quotes stripped — proving the corruption occurred in the shell-to-executable handoff, not in the file or the read. The same payload passed through the batch-input and standard-input mechanisms, which bypass that handoff, was received intact and validated successfully.

The fix follows directly from the diagnosis. Configuration writes that carry structured data must not pass that data as a command-line argument on this platform. They must use standard input or a batch-input file, both of which deliver the payload to the executable without the intervening argument-parsing step. This is consistent with the mechanism already used, successfully and without incident, by every other configuration write in the system; the faulty write was the lone exception, and correcting it means bringing it into line with the rest.

A further observation sharpens the diagnosis and clarifies the correct remedy. The host's default shell was an older major version, the one that ships with the operating system. A newer major version of the same shell, installed separately, changes how native-command arguments are passed and does not exhibit the corruption. This is a genuine difference, but it is the wrong lever to pull. Requiring the newer shell would fix the symptom for a user who installs it while leaving the underlying fragility in the code, and it would exclude every user who runs the shell that ships with the operating system — which is the environment the entire system targets. The correct remedy is the one that does not depend on shell version at all: deliver structured payloads through standard input or a file, which is robust under every shell version because it never crosses the argument-parsing boundary where the corruption occurs. The newer shell is a reasonable recommendation; it is not an acceptable requirement, because a requirement papers over a defect the code should simply not have.

The corrected write also does more than write correctly going forward. Because a prior corrupted run could have left the model list in a damaged state — a duplicated entry, arising when the malformed payload was treated as a new record rather than an update — the repaired routine reads the current list, collapses any duplicates by identity, edits the target entry in place so that all fields it does not touch are preserved intact, and writes the whole reconciled list back through the safe channel. This converts the fix from a write into a repair: it not only applies the intended values but heals a configuration that an earlier failure may already have corrupted. Preserving untouched fields in place is important for one field in particular — the flag that determines whether the model is offered tools — whose loss produces the prose-narration failure described in the appendix.

We note one point of process, because it bears on the paper's own reliability. During diagnosis, an explanation was recorded that attributed the duplication to the merge semantics of the write command — the theory that the command failed to merge entries by identity and therefore appended a duplicate. That explanation is incorrect: the duplication, where it occurred, was a downstream consequence of the shell corrupting the payload's identity field before the executable ever parsed it, not a defect in the merge logic, which the platform's documentation describes and which behaves as documented. The distinction matters beyond pedantry. A recorded root cause becomes an invariant that guides future work, and an incorrect one guides it wrongly; a note blaming the merge command would lead a later reader to distrust a mechanism that is in fact sound, and to avoid it in favor of workarounds that address a cause that was never real. We correct it here for the same reason we recount the two earlier wrong hypotheses: the value of the record lies in its accuracy about why, not merely that, the fix works.

## 7. Lessons

Several general lessons emerge from this work, stated here in the hope they transfer beyond the specific components involved.

**Primary documentation outranks secondary summaries, and generated summaries most of all.** At multiple points we were offered confident, specific, and incorrect claims — about model specifications, about configuration schema, about command semantics — from summaries that carried the surface texture of authority. In each case the resolution was to consult the primary source: the model's own metadata, the schema emitted by the installed binary, the platform's own command reference. On one occasion a summary described an entire subsystem that did not exist in the software, apparently drawn from unrelated projects. The discipline that served us was to treat any claim not traceable to a primary source as a hypothesis, not a fact.

**On constrained hardware, verify residency, not acceptance.** A configuration value being accepted, a parameter being set, a command reporting success — none of these establish that the intended state was achieved on the hardware. The runtime allocation was frequently "correct" in the configuration file while the loaded model ran at the wrong value, because a resident model does not track configuration changes. The only trustworthy signal was direct inspection of what was actually loaded and where it was running.

**Silent failures are the expensive ones.** Every failure that cost significant time shared the property of reporting success, or of raising no error, while failing to achieve its effect. The repair step that discarded changes while saving them aside; the configuration write that reported success while the shell had corrupted its payload; the context divergence that truncated prompts with nothing in any log. Loud failures are cheap to fix. The engineering value lies in converting silent failures into loud ones — which is the purpose of the diagnostic suite the system carries, whose ordering is designed to distinguish failures that present identically from the outside.

**Structured data and command-line arguments are an unsafe combination across shell boundaries.** The specific bug at the center of this paper generalizes: passing JSON, or any quote-bearing structured payload, as a command-line argument through a shell to a native executable is fragile, and the fragility is platform- and shell-dependent in ways that are easy to miss because the same command may work under one shell and fail under another. Standard input and file-based input avoid the problem entirely. This is worth adopting as a default rather than a remedy.

**Capability is rarely the bottleneck; plumbing is.** The recurring temptation, when an agent misbehaves, is to blame the model. In our experience the model was almost never the problem. The problems were a context cap that never applied, a tool interface that was never offered because a field was stripped during a merge, a device that had not finished booting, a skill that never loaded because of an encoding artifact. These are plumbing failures, and they present to the user as if the model were incapable. Diagnosing the plumbing before questioning the model is the more efficient order of operations, and it is the order the system's diagnostic suite enforces.

## 8. Conclusion

We have described a working local agentic system on a 12 GB consumer GPU and, more usefully, a catalog of the failures encountered in building it. The system runs a vision-language model resident entirely on the GPU at a capped context, drives an emulated device through a tool bridge, and is controlled over a messaging channel — all locally. The engineering that made this possible was overwhelmingly concerned not with the model but with the correct, verified propagation of a few coupled numeric parameters through a stack of layers that each had their own idea of what those parameters should be.

With the configuration-write defect corrected, the system reached and held the intended state: the model resident entirely on the GPU, at the capped context, with its tool and vision capabilities intact, verified by direct inspection of the loaded model rather than inferred from the configuration file. That last clause is the discipline the whole exercise teaches in miniature — the configuration had, at various points, been correct while the running system was not, and only inspection of the running system could tell the difference.

The most transferable result is methodological. On hardware below the comfortable tier, the difference between a working system and a mysteriously degraded one lies in a small number of quantities that must be held in agreement and verified at the point of hardware allocation rather than the point of configuration. The failures that matter are silent, and the primary engineering activity is making them loud. We offer this record so that the next person to attempt a similar system on similar hardware inherits the diagnoses rather than repeating them.

## Appendix: Reproducible Findings

The following findings were each encountered directly, diagnosed against primary sources, and resolved. They are stated in general terms so as to transfer across specific tool versions.

A weak or improperly configured model that cannot emit structured tool calls will instead describe, in prose, the action it would take — often inventing plausible but nonexistent command syntax. This presents identically to a model that was never offered tools and to a device that is absent. Distinguishing these three requires a diagnostic that checks each independently and in the right order.

A configuration file rendered invalid causes the repair utility to silently restore a prior copy and discard pending changes. Validation must precede repair.

Updating a protected list-valued configuration path by replacing the whole list discards sibling entries and strips fields that were not restated. Two fields are especially easy to lose this way: the flag governing whether a model is offered tools, whose loss produces the prose-narration failure above, and the flag declaring the model's accepted input modalities, whose loss silently removes image input and blinds a vision agent. We observed both: a minimal hand-authored entry, written to clamp context, dropped the image-input modality and left the model text-only. The remedy is to read the existing entry, edit only the intended fields, and write the whole entry back, so that every field not deliberately changed is preserved verbatim.

The context parameters at different layers must be held in agreement or prompts are silently truncated. The repair utility actively pushes one of them out of agreement and must be sequenced against.

A model already resident in memory does not adopt configuration changes until evicted and reloaded. A correct configuration can therefore coexist with an incorrectly loaded model, and diagnostics that inspect only the loaded model will report failure on a correctly configured system.

Passing structured data as a command-line argument through a shell to a native executable risks corruption of that data at the shell boundary, in a manner dependent on the specific shell and even on its major version. Standard input and file-based input avoid this entirely and are robust across versions. On the platform studied, the shell that ships with the operating system stripped quotation marks from a JSON argument, silently defeating the single configuration write that used an argument rather than standard input; a separately installed newer version of the same shell did not exhibit the corruption. The version difference is a trap, not a fix: relying on it would leave the fragility in the code and exclude users of the default shell. The durable remedy is to never pass structured data as an argument, regardless of shell version.

Files written by naive redirection or by certain default text-encoding routines acquire byte-order marks or wide encodings that downstream parsers reject. All generated files must be written in the encoding the consuming tool expects, without a byte-order mark where one would break parsing.

A device-control bridge reports a device as attached well before that device has finished initializing. Acting on the device in that window fails in ways indistinguishable from a broken tool. Readiness must be polled on an explicit signal, not inferred from attachment, and never waited on with a mechanism that lacks a timeout.

Rendering an emulated device's display in software — commonly recommended as a way to conserve discrete-GPU memory for the model — produced a blank framebuffer on the host studied: the guest booted and its compositor ran, but no pixels were emitted, defeating any agent that must perceive the screen. The failure was identical under software rendering and under hardware rendering forced onto the discrete GPU, which distinguishes it from a mere driver preference. The durable solution rendered in hardware on the host's *integrated* GPU, pinned there through the operating system's per-application GPU preference; this produced a correct image while leaving the discrete GPU's memory entirely for the model. The diagnosis depended on capturing the framebuffer directly and observing that it was near-empty, rather than trusting that a booted device was a visible one — the same "verify the running state, not the reported state" discipline that governs the memory-residency check.
